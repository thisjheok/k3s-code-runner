# Go API 서버 구축하기: 코드 실행 요청을 Kubernetes 작업으로 연결하기

## 들어가며

1편에서는 VM 3대를 기반으로 k3s 클러스터를 구성하고, Mini Code Runner를 실행할 Kubernetes 기반을 마련한 과정을 정리했습니다.

이번 글에서는 그 위에서 동작하는 Go API 서버를 다룹니다. 이 API 서버는 사용자가 제출한 코드를 직접 실행하지 않습니다. 대신 실행 요청을 검증하고, Redis에 실행 정보를 저장한 뒤, worker가 처리할 수 있도록 queue에 넣습니다. 또한 Kubernetes API를 조회해 실행 상태, 로그, worker 상태, node 분산 현황을 사용자에게 보여줍니다.

즉, 이번 프로젝트의 Go API 서버는 단순한 CRUD 서버가 아니라, 브라우저 UI와 Redis queue, Kubernetes Job 실행 구조를 연결하는 진입점입니다.

## 왜 API 서버를 만들었는가

Mini Code Runner의 목적은 사용자가 브라우저에서 코드를 제출하고, 그 코드가 Kubernetes 위에서 독립적인 Job으로 실행되는 흐름을 만드는 것이었습니다.

처음부터 중요하게 생각한 점은 API 서버가 사용자 코드를 직접 실행하지 않도록 하는 것이었습니다. API 서버 프로세스 안에서 코드를 실행하면 구현은 단순해질 수 있습니다. 하지만 실행 시간이 길어지는 요청, 무한 루프, 메모리 사용량 증가, 여러 요청의 병렬 처리, 실행 로그 수집 같은 문제를 API 서버가 직접 떠안게 됩니다.

그래서 API 서버의 역할을 다음처럼 제한했습니다.

- 사용자 요청을 HTTP API로 받습니다.
- 실행 언어와 코드 내용을 검증합니다.
- 실행 요청을 Redis에 저장하고 queue에 넣습니다.
- 실행 상태와 로그를 조회할 수 있는 API를 제공합니다.
- Kubernetes Job, Pod, worker 상태를 읽어 UI에 전달합니다.

코드 실행 자체는 worker와 Kubernetes Job이 담당합니다. API 서버는 실행을 조율하고 상태를 보여주는 역할에 집중하도록 설계했습니다.

## 전체 구조

Go API 서버는 다음 구성 요소와 연결됩니다.

```text
Browser UI
  -> Go API Server
  -> Redis
     - run:{run_id}
     - runs:recent
     - runs:queue

Go API Server
  -> Kubernetes API
     - Job 조회
     - Pod 조회
     - Pod 로그 조회
     - worker Deployment 조회
```

사용자가 코드를 제출하면 API 서버는 Redis에 실행 정보를 저장합니다. worker는 Redis queue에서 `run_id`를 꺼내 Kubernetes Job을 생성합니다. API 서버는 이후 Kubernetes API를 조회해 Job과 Pod 상태를 가져오고, 사용자는 UI에서 실행 결과를 확인할 수 있습니다.

이 구조에서 API 서버는 쓰기 흐름과 읽기 흐름을 동시에 담당합니다.

쓰기 흐름은 다음과 같습니다.

```text
POST /runs
  -> 요청 검증
  -> run_id 생성
  -> Redis hash에 실행 정보 저장
  -> Redis list에 run_id enqueue
  -> 201 Created 반환
```

읽기 흐름은 다음과 같습니다.

```text
GET /runs/{run_id}
  -> Kubernetes Job 조회
  -> Job이 아직 없으면 Redis record 조회
  -> 실행 상태 반환

GET /runs/{run_id}/logs
  -> runner Pod 조회
  -> Pod 로그 stream 조회
  -> 로그 반환
```

## 애플리케이션 시작 흐름

API 서버의 시작 지점은 `cmd/api/main.go`입니다.

서버가 시작되면 먼저 설정을 로드하고, Kubernetes client와 Redis store를 준비합니다.

```go
cfg := config.Load()
logger := slog.New(slog.NewTextHandler(os.Stdout, &slog.HandlerOptions{Level: slog.LevelInfo}))

clientset, err := k8s.NewClientset(cfg.KubeconfigPath)
store := runs.NewRedisStore(cfg.RedisAddr, cfg.RedisQueueName, cfg.RedisRecentRunsKey)
```

Redis는 서버 시작 시점에 `PING`으로 연결 가능 여부를 확인합니다.

```go
if err := store.Ping(context.Background()); err != nil {
    logger.Error("failed to connect to redis", "err", err)
    os.Exit(1)
}
```

이렇게 한 이유는 API 서버가 Redis 없이 정상적으로 동작할 수 없기 때문입니다. 실행 요청 저장과 queue 처리가 모두 Redis에 의존하므로, 시작 단계에서 실패를 빠르게 드러내는 편이 낫다고 판단했습니다.

Kubernetes client와 Redis store가 준비되면 `runs.Service`를 만들고, 이 서비스를 HTTP handler에 주입합니다.

```go
service := runs.NewService(clientset, store, cfg, logger)
handler := api.NewHandler(service, logger)
```

이 구조 덕분에 HTTP handler는 요청과 응답 처리에 집중하고, 실행 요청의 핵심 로직은 service 레이어에서 처리할 수 있습니다.

## 설정 관리

설정은 `internal/config/config.go`에서 환경 변수 기반으로 로드합니다.

```go
type Config struct {
    Port               string
    Namespace          string
    KubeconfigPath     string
    DefaultTimeoutSecs int64
    TTLSeconds         int32
    NodeSelector       map[string]string
    RedisAddr          string
    RedisQueueName     string
    RedisRecentRunsKey string
}
```

주요 설정은 다음과 같습니다.

| 환경 변수 | 기본값 | 설명 |
| --- | --- | --- |
| `PORT` | `8080` | API 서버 포트입니다. |
| `RUNNER_NAMESPACE` | `code-runner-system` | Job과 Pod를 생성하거나 조회할 namespace입니다. |
| `KUBECONFIG` | 빈 값 | Pod 밖에서 실행할 때 사용할 kubeconfig 경로입니다. |
| `RUNNER_TIMEOUT_SECONDS` | `10` | 실행 요청의 기본 timeout입니다. |
| `RUNNER_TTL_SECONDS` | `300` | 완료된 Job을 유지할 시간입니다. |
| `RUNNER_NODE_SELECTOR` | 빈 값 | runner Pod를 배치할 node selector입니다. |
| `REDIS_ADDR` | `code-runner-redis:6379` | Redis 주소입니다. |
| `REDIS_QUEUE_NAME` | `runs:queue` | 실행 대기 queue 이름입니다. |
| `REDIS_RECENT_RUNS_KEY` | `runs:recent` | 최근 실행 목록 key입니다. |

특히 `RUNNER_NODE_SELECTOR`는 `runner=true` 같은 문자열을 map으로 파싱합니다.

```go
NodeSelector: parseNodeSelector(getEnv("RUNNER_NODE_SELECTOR", ""))
```

이 설정은 이후 worker가 Kubernetes Job을 만들 때 runner Pod의 `nodeSelector`로 사용됩니다.

## Kubernetes client 구성

API 서버는 Kubernetes API를 직접 조회해야 합니다. 이를 위해 `internal/k8s/client.go`에서 client-go 기반 clientset을 생성합니다.

```go
cfg, err := rest.InClusterConfig()
if err == nil {
    return kubernetes.NewForConfig(cfg)
}
```

먼저 in-cluster config를 시도합니다. API 서버가 Kubernetes Pod 안에서 실행될 때는 ServiceAccount token과 cluster CA를 이용해 Kubernetes API에 접근할 수 있습니다.

만약 Pod 밖에서 실행 중이라면 kubeconfig를 사용합니다.

```go
cfg, err = clientcmd.BuildConfigFromFlags("", path)
```

이 fallback 구조는 개발할 때 유용했습니다. 로컬 또는 VM에서 직접 `go run ./cmd/api`로 서버를 실행해도 `~/.kube/config`를 통해 k3s에 접근할 수 있기 때문입니다.

즉, 같은 코드를 클러스터 내부와 외부 개발 환경에서 모두 실행할 수 있도록 만든 것입니다.

## HTTP API 설계

HTTP route는 `internal/api/handler.go`에 정의했습니다.

```go
mux.HandleFunc("GET /health", h.health)
mux.HandleFunc("GET /system/status", h.systemStatus)
mux.HandleFunc("POST /runs", h.createRun)
mux.HandleFunc("GET /runs", h.listRuns)
mux.HandleFunc("GET /runs/{run_id}", h.getRun)
mux.HandleFunc("GET /runs/{run_id}/logs", h.getLogs)
mux.HandleFunc("DELETE /runs/{run_id}", h.deleteRun)
```

Go 표준 라이브러리의 `http.ServeMux`를 사용했습니다. 별도 web framework를 사용하지 않은 이유는 API 범위가 작고, 복잡한 middleware나 routing 기능이 필요하지 않았기 때문입니다.

API는 다음 역할로 나눌 수 있습니다.

| API | 역할 |
| --- | --- |
| `GET /health` | 서버 상태 확인 |
| `POST /runs` | 코드 실행 요청 생성 |
| `GET /runs` | 최근 실행 목록 조회 |
| `GET /runs/{run_id}` | 특정 실행 상태 조회 |
| `GET /runs/{run_id}/logs` | 실행 Pod 로그 조회 |
| `DELETE /runs/{run_id}` | 실행 관련 리소스 삭제 |
| `GET /system/status` | queue, worker, runner Pod 상태 조회 |

또한 `/` 경로와 `/static/` 경로를 통해 브라우저 UI도 함께 제공합니다.

```go
mux.HandleFunc("GET /{$}", h.index)
mux.Handle("GET /static/", http.StripPrefix("/static/", http.FileServer(http.Dir("web"))))
```

이번 프로젝트에서는 API 서버와 정적 UI를 하나의 컨테이너 이미지에 담았습니다. 학습용 프로젝트에서는 배포 단위를 줄이고 구조를 빠르게 확인하는 것이 중요했기 때문에 이 방식이 적합했습니다.

## 실행 요청 생성 흐름

가장 중요한 API는 `POST /runs`입니다.

요청 예시는 다음과 같습니다.

```bash
curl -X POST http://code-runner.local/runs \
  -H "Content-Type: application/json" \
  -d '{"language":"python","code":"print(\"hello\")","timeout_seconds":30}'
```

handler는 JSON body를 `CreateRequest`로 decode한 뒤 service에 전달합니다.

```go
var req runs.CreateRequest
if err := json.NewDecoder(r.Body).Decode(&req); err != nil {
    writeError(w, http.StatusBadRequest, "invalid JSON body")
    return
}

resp, err := h.runs.Create(r.Context(), req)
```

실제 생성 로직은 `runs.Service.Create`에 있습니다.

```go
func (s *Service) Create(ctx context.Context, req CreateRequest) (RunResponse, error) {
    if _, err := getLanguageSpec(req.Language); err != nil {
        return RunResponse{}, err
    }
    if strings.TrimSpace(req.Code) == "" {
        return RunResponse{}, fmt.Errorf("%w: code is required", ErrInvalidInput)
    }

    timeout := req.TimeoutSeconds
    if timeout <= 0 {
        timeout = s.cfg.DefaultTimeoutSecs
    }

    runID, err := newRunID()
    ...
}
```

여기서 API 서버는 다음을 검증합니다.

- 지원하는 언어인지 확인합니다.
- 코드가 비어 있지 않은지 확인합니다.
- timeout이 없으면 기본값을 사용합니다.
- 실행 요청을 식별할 `run_id`를 생성합니다.

지원 언어는 현재 Python과 Node.js입니다.

```go
case "", "python", "py":
    return languageSpec{
        Image:   "python:3.11-slim",
        File:    "main.py",
        Command: []string{"python", "/workspace/main.py"},
    }, nil
case "node", "javascript", "js":
    return languageSpec{
        Image:   "node:20-slim",
        File:    "main.js",
        Command: []string{"node", "/workspace/main.js"},
    }, nil
```

언어별 실행 이미지, 파일명, command를 API 서버가 알고 있는 이유는 이후 worker가 Job을 만들 때 같은 실행 스펙을 사용해야 하기 때문입니다.

## Redis에 실행 요청 저장하기

검증이 끝나면 API 서버는 실행 요청을 Redis에 저장합니다.

```go
record := RunRecord{
    RunID:          runID,
    Language:       strings.TrimSpace(req.Language),
    Code:           req.Code,
    TimeoutSeconds: timeout,
    Status:         statusQueued,
}

if err := s.store.EnqueueRun(ctx, record); err != nil {
    return RunResponse{}, err
}
```

Redis에는 세 가지 형태로 데이터가 저장됩니다.

| Redis key | 역할 |
| --- | --- |
| `run:{run_id}` | 실행 언어, 코드, timeout, 상태, 생성 시각을 저장합니다. |
| `runs:recent` | 최근 실행 목록을 보여주기 위한 list입니다. |
| `runs:queue` | worker가 소비할 실행 대기 queue입니다. |

저장은 Redis transaction pipeline으로 처리합니다.

```go
pipe := s.client.TxPipeline()
pipe.HSet(ctx, key, map[string]any{
    "run_id":          record.RunID,
    "language":        record.Language,
    "code":            record.Code,
    "timeout_seconds": strconv.FormatInt(record.TimeoutSeconds, 10),
    "status":          record.Status,
    "created_at":      record.CreatedAt,
})
pipe.LPush(ctx, s.recentRunsKey, record.RunID)
pipe.LTrim(ctx, s.recentRunsKey, 0, 99)
pipe.RPush(ctx, s.queueName, record.RunID)
_, err := pipe.Exec(ctx)
```

이렇게 구성한 이유는 실행 요청 생성 시점에 필요한 작업을 하나의 흐름으로 묶기 위해서입니다.

실행 상세 정보는 hash에 저장하고, 최근 목록은 `runs:recent`에 보관합니다. worker 처리를 위한 queue는 `runs:queue`에 저장합니다. API 서버는 실행 요청을 Redis에 넣은 뒤 바로 응답하고, 실제 Kubernetes Job 생성은 worker가 비동기로 처리합니다.

이 구조 덕분에 API 서버는 긴 코드 실행 시간을 기다리지 않습니다.

## 왜 바로 Job을 만들지 않았는가

처음에는 API 서버가 `POST /runs` 요청을 받자마자 Kubernetes Job을 생성하는 방식도 생각할 수 있습니다. 실제로 구조만 단순하게 보면 이 방식이 더 직관적입니다.

하지만 이번 프로젝트에서는 API 서버와 Job 생성을 분리했습니다.

가장 큰 이유는 오토스케일링을 실험하기 위해서입니다. 실행 요청이 Redis queue에 쌓이고, worker가 queue를 소비하는 구조가 있어야 KEDA가 queue 길이를 기준으로 worker replica 수를 조절할 수 있습니다.

또한 API 서버와 worker의 책임을 나누면 다음 장점이 있습니다.

- API 서버는 HTTP 요청 처리와 상태 조회에 집중할 수 있습니다.
- worker는 queue 소비와 Job 생성에 집중할 수 있습니다.
- 실행 요청이 몰려도 API 서버가 직접 모든 Job 생성을 처리하지 않습니다.
- Redis queue 길이를 시스템 부하 지표로 사용할 수 있습니다.
- KEDA를 붙여 worker를 0에서 N개까지 자동 확장할 수 있습니다.

따라서 API 서버는 실행 요청을 “접수”하고, worker는 실행 요청을 “처리”하는 구조로 나누었습니다.

## 실행 상태 조회

`GET /runs/{run_id}`는 실행 상태를 조회하는 API입니다.

흥미로운 점은 상태 조회 시 Kubernetes Job을 먼저 확인한다는 것입니다.

```go
job, err := s.clientset.BatchV1().Jobs(s.cfg.Namespace).Get(ctx, resourceName(runID), metav1.GetOptions{})
```

Job이 존재하면 Job과 Pod 상태를 기반으로 응답을 만듭니다.

```go
return s.detailFromJob(ctx, runID, job)
```

반대로 Job이 아직 없다면 Redis record를 조회합니다.

```go
if apierrors.IsNotFound(err) {
    record, recordErr := s.store.GetRun(ctx, runID)
    ...
    return detailFromRecord(record), nil
}
```

이 방식은 queue 기반 구조와 잘 맞습니다.

사용자가 `POST /runs`를 호출한 직후에는 worker가 아직 Job을 만들지 않았을 수 있습니다. 이때 Kubernetes에는 Job이 없지만 Redis에는 `Queued` 상태의 실행 요청이 존재합니다. 따라서 API 서버는 Job이 없다고 바로 404를 반환하지 않고, Redis record를 확인해 queue 상태를 알려줍니다.

상태는 Job condition과 Pod 상태를 조합해 계산합니다.

```go
if pod.Status.Phase == corev1.PodRunning {
    return "Running"
}

for _, condition := range job.Status.Conditions {
    if condition.Type == batchv1.JobComplete && condition.Status == corev1.ConditionTrue {
        return "Succeeded"
    }
    if condition.Type == batchv1.JobFailed && condition.Status == corev1.ConditionTrue {
        if condition.Reason == "DeadlineExceeded" {
            return "Timeout"
        }
        return statusFailed
    }
}
return "Pending"
```

이렇게 해서 UI는 단순히 Kubernetes 원본 상태를 보여주는 대신, 사용자에게 이해하기 쉬운 상태값을 받을 수 있습니다.

## 로그 조회

`GET /runs/{run_id}/logs`는 runner Pod의 로그를 조회합니다.

먼저 실행 요청에 해당하는 Pod를 label로 찾습니다.

```go
pods, err := s.clientset.CoreV1().Pods(s.cfg.Namespace).List(ctx, metav1.ListOptions{
    LabelSelector: labels.SelectorFromSet(labels.Set{labelRunID: runID}).String(),
})
```

Pod가 존재하면 Kubernetes Pod log API를 호출합니다.

```go
reader, err := s.clientset.CoreV1().Pods(s.cfg.Namespace).GetLogs(pod.Name, &corev1.PodLogOptions{
    Container: "runner",
}).Stream(ctx)
```

만약 아직 worker가 Job을 만들지 않아 Pod가 없다면 다음 메시지를 반환합니다.

```text
Run is queued. Logs will be available after a worker creates the Kubernetes Job.
```

이 처리는 사용자 경험 측면에서 중요했습니다. queue 기반 구조에서는 실행 요청 직후 로그가 없는 것이 정상입니다. 이때 단순히 에러를 반환하면 사용자는 실행이 실패했다고 오해할 수 있습니다. 그래서 아직 queue에 있다는 상태를 명확히 알려주도록 했습니다.

## 시스템 상태 API

브라우저 UI의 Live Monitor는 `GET /system/status`를 주기적으로 호출합니다.

이 API는 단일 실행 요청이 아니라 시스템 전체 상태를 보여주기 위해 만들었습니다.

응답에는 다음 정보가 포함됩니다.

```go
type SystemStatusResponse struct {
    QueueName           string         `json:"queue_name"`
    QueueDepth          int64          `json:"queue_depth"`
    QueuedRunIDs        []string       `json:"queued_run_ids"`
    WorkerReplicas      int32          `json:"worker_replicas"`
    WorkerReadyReplicas int32          `json:"worker_ready_replicas"`
    WorkerPods          []WorkerPod    `json:"worker_pods"`
    RunnerJobs          []RunnerJob    `json:"runner_jobs"`
    RunnerPods          []RunnerPod    `json:"runner_pods"`
    NodeRuns            map[string]int `json:"node_runs"`
}
```

API 서버는 Redis에서 queue 길이를 조회합니다.

```go
queueDepth, err := s.store.QueueDepth(ctx)
queuedRunIDs, err := s.store.QueuedRunIDs(ctx, 50)
```

그리고 Kubernetes에서 worker Deployment와 worker Pod를 조회합니다.

```go
deployment, err := s.clientset.AppsV1().Deployments(s.cfg.Namespace).Get(ctx, "code-runner-worker", metav1.GetOptions{})

workerPods, err := s.clientset.CoreV1().Pods(s.cfg.Namespace).List(ctx, metav1.ListOptions{
    LabelSelector: labels.SelectorFromSet(labels.Set{"app": "code-runner-worker"}).String(),
})
```

runner Pod와 Job도 함께 조회합니다.

```go
runnerPods, err := s.clientset.CoreV1().Pods(s.cfg.Namespace).List(ctx, metav1.ListOptions{
    LabelSelector: labels.SelectorFromSet(labels.Set{labelManagedBy: "mini-code-runner"}).String(),
})

jobs, err := s.clientset.BatchV1().Jobs(s.cfg.Namespace).List(ctx, metav1.ListOptions{
    LabelSelector: labels.SelectorFromSet(labels.Set{labelManagedBy: "mini-code-runner"}).String(),
})
```

이 API를 만든 이유는 KEDA 기반 오토스케일링을 관찰하기 위해서입니다. queue가 쌓이고, worker replica가 증가하고, runner Pod가 여러 node에 배치되는 흐름을 UI에서 한 번에 볼 수 있어야 했습니다.

특히 node별 runner Pod 수를 계산하는 `nodeRuns`는 멀티 노드 k3s 클러스터를 구성한 이유를 확인하는 지표입니다.

```go
nodeRuns := map[string]int{}
...
nodeRuns[nodeName]++
```

이를 통해 실행 요청이 어느 node에서 처리되고 있는지 확인할 수 있습니다.

## 리소스 삭제

`DELETE /runs/{run_id}`는 실행 요청과 관련된 리소스를 정리합니다.

삭제 대상은 세 가지입니다.

- Kubernetes Job
- 코드가 저장된 ConfigMap
- Redis에 저장된 실행 record와 queue/recent list 항목

```go
jobErr := s.clientset.BatchV1().Jobs(s.cfg.Namespace).Delete(ctx, name, metav1.DeleteOptions{
    PropagationPolicy: &propagation,
})
cmErr := s.clientset.CoreV1().ConfigMaps(s.cfg.Namespace).Delete(ctx, name, metav1.DeleteOptions{})
storeErr := s.store.DeleteRun(ctx, runID)
```

Job 삭제에는 background propagation을 사용했습니다. Job이 생성한 Pod까지 Kubernetes가 자연스럽게 정리할 수 있도록 하기 위해서입니다.

```go
propagation := metav1.DeletePropagationBackground
```

학습용 프로젝트라도 생성되는 리소스가 많아질 수 있으므로, 삭제 API를 통해 실험 중 만든 실행 리소스를 정리할 수 있게 했습니다.

## 에러 처리 방식

handler에서는 입력 오류와 서버 내부 오류를 구분했습니다.

예를 들어 지원하지 않는 언어이거나 코드가 비어 있으면 `400 Bad Request`를 반환합니다.

```go
if errors.Is(err, runs.ErrInvalidInput) {
    writeError(w, http.StatusBadRequest, err.Error())
    return
}
```

반면 Redis나 Kubernetes API 호출 실패처럼 서버 내부 문제가 발생하면 `500 Internal Server Error`를 반환합니다.

```go
h.logger.Error("create run failed", "err", err)
writeError(w, http.StatusInternalServerError, "failed to create run")
```

응답 형식은 JSON으로 통일했습니다.

```go
func writeError(w http.ResponseWriter, status int, message string) {
    writeJSON(w, status, map[string]string{"error": message})
}
```

작은 API 서버라도 에러 응답 형식을 통일해두면 UI에서 처리하기 쉽고, 테스트할 때도 결과를 이해하기 쉽습니다.

## graceful shutdown

API 서버는 `SIGINT`, `SIGTERM`을 받아 graceful shutdown을 수행합니다.

```go
ctx, stop := signal.NotifyContext(context.Background(), syscall.SIGINT, syscall.SIGTERM)
defer stop()

<-ctx.Done()
shutdownCtx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
defer cancel()

if err := server.Shutdown(shutdownCtx); err != nil {
    logger.Error("server shutdown failed", "err", err)
    os.Exit(1)
}
```

Kubernetes에서 Pod가 종료될 때 바로 프로세스를 끊는 대신, 기존 요청을 정리할 시간을 주기 위한 처리입니다.

이번 프로젝트에서는 API 서버가 장시간 요청을 직접 처리하지 않지만, 배포와 재시작이 반복되는 환경에서는 graceful shutdown을 기본으로 넣어두는 편이 좋다고 판단했습니다.

## 컨테이너 이미지 구성

Dockerfile은 multi-stage build로 구성했습니다.

```dockerfile
FROM golang:1.23-bookworm AS builder

WORKDIR /src
COPY go.mod go.sum* ./
RUN go mod download

COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -o /out/code-runner-api ./cmd/api
RUN CGO_ENABLED=0 GOOS=linux go build -o /out/code-runner-worker ./cmd/worker
```

최종 이미지는 distroless 기반입니다.

```dockerfile
FROM gcr.io/distroless/static-debian12:nonroot
WORKDIR /
COPY --from=builder /out/code-runner-api /code-runner-api
COPY --from=builder /out/code-runner-worker /code-runner-worker
COPY --from=builder /src/web /web
EXPOSE 8080
ENTRYPOINT ["/code-runner-api"]
```

하나의 이미지에 API 서버와 worker 바이너리를 함께 넣고, Deployment에서 command를 다르게 지정했습니다.

API 서버는 기본 entrypoint를 사용합니다.

```yaml
containers:
  - name: api
    image: code-runner-api:latest
```

worker는 command를 override합니다.

```yaml
containers:
  - name: worker
    image: code-runner-api:latest
    command:
      - /code-runner-worker
```

이 방식은 이미지 빌드와 배포를 단순하게 만듭니다. API와 worker가 같은 코드베이스와 설정 구조를 공유하고 있으므로, 프로젝트 규모에서는 하나의 이미지로 관리하는 편이 더 실용적이었습니다.

## 설계하며 고민한 점

이번 API 서버에서 가장 중요한 설계 판단은 실행 요청 처리와 실제 실행을 분리한 것입니다.

API 서버가 직접 Job을 만들 수도 있었지만, Redis queue를 중간에 둔 덕분에 실행 요청을 비동기로 처리할 수 있었습니다. 또한 queue 길이라는 명확한 지표가 생겼고, 이를 기반으로 KEDA가 worker를 자동 확장할 수 있는 구조를 만들 수 있었습니다.

두 번째는 Kubernetes 상태를 API 서버에서 가공해 UI에 제공한 것입니다.

Kubernetes의 Job, Pod, Deployment 상태를 그대로 보여주면 정보는 많지만 사용자 입장에서는 이해하기 어렵습니다. 그래서 API 서버가 `Queued`, `Running`, `Succeeded`, `Failed`, `Timeout`, `OOMKilled` 같은 상태로 정리해 반환하도록 했습니다.

세 번째는 표준 라이브러리 중심으로 단순하게 구현한 것입니다.

이번 프로젝트의 핵심은 web framework 사용법이 아니라 Kubernetes Job 기반 실행 구조였습니다. 그래서 routing은 `net/http`의 `ServeMux`로 충분하다고 판단했고, 외부 의존성은 Redis client와 Kubernetes client-go 중심으로 제한했습니다.

## 마무리

이번 글에서는 Mini Code Runner의 Go API 서버를 살펴봤습니다.

API 서버는 사용자의 코드 실행 요청을 받아 Redis에 저장하고, worker가 처리할 수 있도록 queue에 넣습니다. 또한 Kubernetes API를 조회해 실행 상태와 로그, worker 상태, node별 실행 분포를 UI에 제공합니다.

이 구조를 통해 API 서버는 직접 코드를 실행하지 않고도 코드 실행 시스템의 진입점 역할을 수행할 수 있었습니다.

이번 편에서 만든 핵심 기반은 다음과 같습니다.

- `POST /runs` 기반 실행 요청 접수
- Redis hash/list를 이용한 실행 record와 queue 관리
- Kubernetes client-go 기반 Job, Pod, Deployment 조회
- Pod log API를 이용한 실행 로그 조회
- `GET /system/status` 기반 queue와 worker 상태 모니터링
- API 서버와 worker를 하나의 이미지로 빌드하는 배포 구조

다음 글에서는 Redis queue와 KEDA를 이용해 worker를 자동으로 확장하는 구조를 다룹니다. 실행 요청이 쌓이면 worker Pod가 늘어나고, queue가 비면 다시 0개로 줄어드는 흐름을 살펴보겠습니다.

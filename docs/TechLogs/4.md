# Kueue 기반 CPU admission control 비교 실험 정리

## 들어가며

앞선 단계에서는 Mini Code Runner에 Redis와 KEDA를 붙여 worker Pod를 자동 확장하는 구조를 만들었습니다.

기존 실행 흐름은 다음과 같았습니다.

```text
POST /runs
-> Redis runs:queue
-> KEDA가 Redis queue 길이를 보고 code-runner-worker scale-out
-> worker Pod가 run_id를 pop
-> worker Pod가 Kubernetes Job 생성
-> runner Pod가 사용자 코드를 실행
```

이 구조는 처음에는 꽤 자연스러워 보였습니다. 실행 요청이 많아지면 Redis queue 길이가 늘어나고, KEDA는 그 queue 길이를 보고 worker Pod 수를 늘립니다. worker는 queue를 빠르게 소비하고 Kubernetes Job을 생성합니다. queue가 비면 KEDA는 worker를 다시 줄입니다.

하지만 부하 실험을 진행하면서 이 구조가 "Job 생성 속도"와 "실제 클러스터 실행 가능량"을 분리해서 보지 못한다는 문제가 드러났습니다.

이번 글에서는 그 문제를 어떻게 발견했고, 왜 Kueue를 대안으로 검토했으며, 어떤 방식으로 기존 구조를 갈아엎지 않고 Kueue를 붙여 비교 실험을 설계했는지 정리합니다.

## 기존 Redis + KEDA 방식의 문제 제기

Redis + KEDA 방식에서 KEDA가 직접 보는 값은 Redis queue 길이입니다.

즉 KEDA의 판단 기준은 다음에 가깝습니다.

```text
runs:queue가 길다
-> worker가 더 필요하다

runs:queue가 비었다
-> worker를 유지할 필요가 없다
```

문제는 worker의 역할이 실제 코드를 실행하는 것이 아니라 Kubernetes Job을 생성하는 것이라는 점입니다.

worker가 충분히 빠르게 queue를 소비하면 Redis queue는 금방 0이 됩니다. KEDA 입장에서는 backlog가 사라진 것처럼 보입니다. 하지만 실제로는 worker가 이미 많은 Kubernetes Job을 만들어 두었고, 그 Job들이 runner Pod로 바뀌어 클러스터 안에서 Running 또는 Pending 상태로 남아 있을 수 있습니다.

실험 중 관찰한 대표적인 상황은 다음과 같았습니다.

```text
Redis queue length = 0
worker replicas = 0
runner Pods = Running 다수, Pending 다수
worker node CPU = 99-100%
```

이 상황은 "queue는 비었지만 클러스터는 이미 포화된 상태"입니다.

특히 기존 80 + 30 부하 실험에서는 queue가 빠르게 비어버린 뒤에도 runner Pod가 대량으로 남았습니다. 기존 방식에서는 Kubernetes scheduler 앞에 Pending Pod가 쌓이는 형태로 backlog가 드러났습니다. 즉 Redis queue가 실행 대기열 역할을 끝까지 유지하지 못하고, 실제 대기열이 Kubernetes Pending Pod 영역으로 밀려난 것입니다.

이 구조의 핵심 문제는 다음과 같습니다.

| 관점 | 기존 Redis + KEDA 방식 |
| --- | --- |
| KEDA가 보는 값 | Redis queue length |
| 실제 병목 위치 | Kubernetes Job/Pod scheduling |
| queue가 0일 때의 오해 | 더 처리할 일이 없다고 보일 수 있음 |
| 실제 클러스터 상태 | runner Pod가 Running/Pending으로 계속 자원을 점유할 수 있음 |
| 부하가 커질 때 | Pending Pod가 폭증하고 scheduler/API server 부담이 커질 수 있음 |

Redis + KEDA 자체가 잘못된 것은 아닙니다. 다만 이 조합은 "worker를 깨우는 장치"로는 적합하지만, "실행 가능한 Job 수를 클러스터 자원 기준으로 제한하는 장치"는 아닙니다.

## 왜 Pending Pod 폭증이 중요한가

Pending Pod가 조금 생기는 것은 Kubernetes에서 흔한 일입니다. 하지만 실험처럼 짧은 시간에 수십 개의 runner Pod가 Pending으로 쌓이는 상황은 단순한 표시상의 문제가 아닙니다.

Pending Pod가 많이 쌓이면 다음 문제가 생길 수 있습니다.

- scheduler가 배치할 수 없는 Pod를 계속 평가해야 합니다.
- Kubernetes API server에 Job/Pod 객체가 급격히 증가합니다.
- 사용자는 "작업이 제출되었다"고 보지만 실제 시작 시간은 예측하기 어려워집니다.
- 새 요청이 들어왔을 때 Redis queue는 비어 있어도 클러스터에는 여유가 없을 수 있습니다.
- 이후 GPU Job으로 확장할 경우 비싼 GPU 자원 앞에서 같은 문제가 더 크게 드러날 수 있습니다.

즉 이번 문제는 단순히 Pending 숫자가 보기 싫은 문제가 아니라, 실행 admission 위치가 잘못되어 있다는 신호에 가깝습니다.

## 해결 방향 고민

처음에는 Prometheus 기반으로 node CPU, memory, runner Running/Pending 수를 관찰해서 worker가 Job을 너무 빨리 만들지 못하게 하는 방식을 생각했습니다.

이 방식은 다음과 같은 형태입니다.

```text
worker가 run_id를 pop하기 전
-> Prometheus 또는 Kubernetes API로 현재 클러스터 상태 확인
-> 여유가 있으면 Job 생성
-> 여유가 없으면 Redis queue에 남겨두거나 재시도
```

이 접근은 직관적입니다. 하지만 직접 구현해야 할 정책이 많습니다.

- 현재 사용률을 어떤 기준으로 볼 것인가
- CPU request와 실제 CPU usage 중 무엇을 우선할 것인가
- Pending Pod 수를 어떻게 반영할 것인가
- 여러 worker가 동시에 판단할 때 race condition을 어떻게 막을 것인가
- Job 공정성, quota, namespace별 우선순위를 어떻게 다룰 것인가

이 고민을 하면서 Kueue를 검토하게 되었습니다.

Kueue는 Kubernetes batch workload를 위한 queueing 및 admission control 도구입니다. 핵심은 Job을 바로 실행시키지 않고, ClusterQueue quota를 기준으로 실행을 허용할 workload만 admit한다는 점입니다.

Kueue를 붙이면 실행 흐름은 다음처럼 바뀝니다.

```text
POST /runs
-> Redis runs:queue
-> KEDA가 worker scale-out
-> worker Pod가 run_id pop
-> worker Pod가 Kueue-managed Job 생성
-> Kueue가 ClusterQueue quota 기준으로 Job admit
-> admitted Job만 runner Pod 생성
-> admit되지 않은 Job은 Kueue Workload queue에서 대기
```

중요한 차이는 backlog의 위치입니다.

기존 방식에서는 backlog가 Kubernetes Pending Pod로 나타났습니다. Kueue 방식에서는 초과 workload가 Kueue Workload queue와 suspended Job으로 남습니다. scheduler가 처리할 수 없는 Pod를 대량으로 만들기 전에, admission 단계에서 실행량을 제한하는 것입니다.

## Kueue가 Prometheus를 완전히 대체하는 것은 아니다

Kueue는 CPU 사용률을 실시간으로 보고 판단하는 도구라기보다, resource request와 quota를 기반으로 "지금 이 Job을 실행시켜도 되는가"를 판단하는 도구입니다.

따라서 역할을 분리해서 보는 것이 좋습니다.

| 구성 요소 | 주 역할 |
| --- | --- |
| Redis | 요청 저장, 최근 실행 기록, 기존 queue |
| KEDA | Redis queue 길이에 따라 worker Deployment scale-out |
| worker | Redis queue를 소비하고 Kubernetes Job 생성 |
| Kueue | Job admission control, quota 기반 실행량 제한 |
| Prometheus | CPU/memory/Pod/Node 상태 관찰, 실험 지표 수집 |
| Karpenter | 이후 노드 자동 확장 및 축소 |

이번 실험에서 Kueue는 Prometheus를 대체하기 위한 것이 아니라, Pending Pod 폭증을 줄이기 위한 admission layer로 도입했습니다.

Prometheus 기반 metric pipeline은 여전히 필요합니다. 특히 최종 목표가 GPU 노드 자동 확장 및 축소라면, GPU 사용률, node idle time, Pending/Running workload, quota 조정 정책 등을 관찰하고 판단하기 위한 metric pipeline은 별도로 유지되어야 합니다.

## 롤백 가능한 도입 방식

이번 단계에서는 Redis + KEDA + worker 구조를 완전히 제거하지 않았습니다. 대신 worker가 생성하는 Job에 Kueue 설정을 선택적으로 붙일 수 있도록 feature flag를 추가했습니다.

추가한 주요 환경 변수는 다음과 같습니다.

```text
KUEUE_ENABLED=false
KUEUE_QUEUE_NAME=code-runner-queue
RUNNER_CPU_REQUEST=100m
RUNNER_MEMORY_REQUEST=64Mi
RUNNER_CPU_LIMIT=500m
RUNNER_MEMORY_LIMIT=256Mi
```

Kueue mode를 켜면 worker는 Job을 다음 방식으로 생성합니다.

```yaml
metadata:
  labels:
    kueue.x-k8s.io/queue-name: code-runner-queue
spec:
  suspend: true
```

여기서 `suspend: true`가 중요했습니다. 단순히 Kueue queue label만 붙이면 기대한 admission 흐름이 나오지 않았습니다. Job을 처음부터 suspend 상태로 만들고, Kueue가 quota를 확인한 뒤 admit하면서 unsuspend해야 runner Pod 생성이 제어됩니다.

Kueue를 끄고 싶으면 다음처럼 worker 환경 변수만 되돌리면 됩니다.

```bash
kubectl -n code-runner-system set env deployment/code-runner-worker KUEUE_ENABLED=false
kubectl -n code-runner-system rollout restart deployment/code-runner-worker
```

Kueue 리소스가 클러스터에 남아 있어도 새 Job에 Kueue label이 붙지 않으면 Kueue가 해당 Job을 관리하지 않습니다. 그래서 이번 도입은 실험 중 문제가 생겨도 기존 Redis + KEDA 방식으로 빠르게 롤백할 수 있습니다.

## Kueue 리소스 설계

CPU 기반 admission control 검증을 위해 다음 Kueue 리소스를 사용했습니다.

```yaml
apiVersion: kueue.x-k8s.io/v1beta1
kind: ResourceFlavor
metadata:
  name: cpu-default
---
apiVersion: kueue.x-k8s.io/v1beta1
kind: ClusterQueue
metadata:
  name: code-runner-cq
spec:
  namespaceSelector: {}
  resourceGroups:
    - coveredResources: ["cpu", "memory"]
      flavors:
        - name: cpu-default
          resources:
            - name: cpu
              nominalQuota: "4"
            - name: memory
              nominalQuota: 4Gi
---
apiVersion: kueue.x-k8s.io/v1beta1
kind: LocalQueue
metadata:
  namespace: code-runner-system
  name: code-runner-queue
spec:
  clusterQueue: code-runner-cq
```

실험에서는 runner CPU request를 `500m`로 올렸습니다.

ClusterQueue CPU quota가 `4`이고 runner 하나가 `500m`를 request하면 이론상 동시에 admit되는 runner Job 수는 다음과 같습니다.

```text
4 CPU / 0.5 CPU = 8개
```

주의할 점은 runner request를 올렸다면 Redis + KEDA baseline도 같은 `RUNNER_CPU_REQUEST=500m`, `RUNNER_CPU_LIMIT=500m` 조건에서 비교해야 한다는 것입니다. 그래야 Kueue 도입 여부만 실험 변수로 남습니다.

## 실험 설계

실험 목표는 다음 두 가지였습니다.

- Kueue 도입 후 runner Pod의 Pending 폭증이 기존 Redis + KEDA 방식보다 줄어드는가
- 초과 workload가 Pending Pod가 아니라 Kueue Workload queue에서 대기하는가

부하 패턴은 기존 문제 상황과 같은 80 + 30 방식으로 잡았습니다.

```text
1. 초기 workload 80개 제출
2. Redis queue가 0에 가까워질 때 추가 workload 30개 제출
3. runner Pod, Kueue Workload, LocalQueue 상태를 관찰
```

주요 관찰 지표는 다음과 같습니다.

| 지표 | 확인 의미 |
| --- | --- |
| Redis queue length | KEDA가 보고 있는 backlog |
| worker replica 수 | KEDA scale-out/scale-down 상태 |
| runner Running Pod 수 | 실제 실행 중인 workload |
| runner Pending Pod 수 | scheduler 앞에서 막힌 workload |
| Job `suspend=true/false` 수 | Kueue admission 전후 Job 수 |
| Kueue Workload `ADMITTED` | quota 기준 admit 여부 |
| LocalQueue pending/admitted workloads | Kueue queue backlog |
| node CPU/memory | 실제 클러스터 자원 포화 여부 |

모니터링은 `tee`로 로그를 남기면서 진행했습니다.

```bash
while true; do
  date
  kubectl -n code-runner-system exec deploy/code-runner-redis -- redis-cli llen runs:queue
  kubectl -n code-runner-system get deploy code-runner-worker
  kubectl -n code-runner-system get pods -l app.kubernetes.io/managed-by=mini-code-runner
  kubectl -n code-runner-system get jobs -l app.kubernetes.io/managed-by=mini-code-runner \
    -o jsonpath='{range .items[*]}{.metadata.name}{" suspend="}{.spec.suspend}{"\n"}{end}'
  kubectl -n code-runner-system get workloads
  kubectl -n code-runner-system get localqueue code-runner-queue
  kubectl get clusterqueue code-runner-cq
  kubectl top nodes
  sleep 5
done | tee kueue-80-30-test.log
```

전체 실행이 끝날 때까지 기다리는 것보다, 다음 패턴이 충분히 관찰될 때까지 보는 것이 핵심입니다.

- Redis queue가 0으로 내려가는 시점
- 추가 30개 workload를 넣은 직후
- LocalQueue pending workload가 증가하는지
- admitted workload 수가 quota 근처에서 제한되는지
- runner Pending Pod가 기존 방식처럼 폭증하지 않는지

## 실험 중 확인한 시행착오

Kueue 리소스를 처음 적용할 때 CRD가 설치되어 있지 않으면 다음과 같은 오류가 납니다.

```text
no matches for kind "ResourceFlavor" in version "kueue.x-k8s.io/v1beta1"
ensure CRDs are installed first
```

이는 Kueue CRD가 아직 설치되지 않았다는 의미입니다. Kueue controller와 CRD를 먼저 설치해야 합니다.

또한 Kueue webhook service endpoint가 준비되기 전에 리소스를 적용하면 webhook 호출 실패가 날 수 있습니다. 이 경우 Kueue controller Pod와 endpoint가 Ready인지 확인한 뒤 다시 적용해야 합니다.

```bash
kubectl -n kueue-system get pods
kubectl -n kueue-system get svc,endpoints
```

또 하나의 중요한 시행착오는 API/worker Deployment의 nodeSelector였습니다. 기존에는 control plane VM에 고정하는 설정이 남아 있었는데, control plane taint 때문에 Pod가 스케줄되지 않는 문제가 있었습니다. 실험을 진행하려면 API와 worker가 실제로 뜰 수 있어야 하므로 해당 nodeSelector를 제거했습니다.

마지막으로 local image를 사용하는 환경에서는 새 image가 모든 worker node에 있어야 합니다. Pod가 worker node에 스케줄되었는데 해당 노드에 image가 없으면 `ImagePullBackOff`가 발생합니다. 이 경우 image를 각 node의 containerd로 import해야 합니다.

## 1차 sanity check 결과

먼저 10개 workload로 Kueue admission이 동작하는지 확인했습니다.

관찰 결과는 다음과 같았습니다.

```text
총 Job 수 = 10
suspend=false = 8
suspend=true = 2
LocalQueue pending workloads = 2
LocalQueue admitted workloads = 8
```

이는 의도한 결과였습니다.

ClusterQueue CPU quota가 `4`, runner CPU request가 `500m`이므로 Kueue는 8개 Job만 admit했고, 나머지 2개는 Kueue queue에서 대기했습니다.

다만 admitted Job 중 일부 runner Pod가 여전히 Pending으로 보이기도 했습니다. 이 점은 Kueue의 quota와 실제 scheduler 배치 가능량이 완전히 동일하지 않을 수 있음을 보여줍니다. Kueue의 admitted는 "quota 관점에서 실행을 허용했다"는 뜻이고, Kubernetes scheduler가 실제 node에 배치했다는 뜻은 아닙니다.

## 80 + 30 Kueue 비교 실험 결과

Kueue mode를 켠 상태에서 80개 workload를 먼저 넣고, queue가 0에 가까워질 때 30개 workload를 추가했습니다.

로그에서 확인한 주요 peak 값은 다음과 같습니다.

| 지표 | Kueue 실험 결과 |
| --- | ---: |
| Redis queue peak | 80 |
| runner Running Pod peak | 5 |
| runner Pending Pod peak | 6 |
| runner ContainerCreating peak | 3 |
| Job `suspend=true` peak | 102 |
| Job `suspend=false` peak | 24 |
| LocalQueue pending workloads peak | 102 |
| LocalQueue admitted workloads peak | 8 |

시간 흐름상 중요한 장면은 다음과 같았습니다.

```text
02:37:05
Redis queue = 80
runner Pod 거의 없음
LocalQueue pending/admitted = 0/0

02:37:09
Redis queue = 71
runner Pending Pod = 6
suspend=false = 8
suspend=true = 6
LocalQueue pending/admitted = 5/8

02:37:22
Redis queue = 0
runner Running Pod = 5
runner Pending Pod = 3
suspend=false = 8
suspend=true = 72
LocalQueue pending/admitted = 72/8

02:37:31
Redis queue = 0
runner Running Pod = 5
runner Pending Pod = 3
suspend=true = 102
LocalQueue pending/admitted = 102/8
```

핵심은 Redis queue가 0이 된 뒤에도 backlog가 사라진 것이 아니라, Kueue LocalQueue에 pending workload로 남았다는 점입니다.

기존 Redis + KEDA baseline에서는 backlog가 Pending Pod로 밀려났습니다. 반면 Kueue 실험에서는 대부분의 backlog가 suspended Job과 Kueue pending workload로 남았습니다.

비교하면 다음과 같습니다.

| 항목 | Redis + KEDA baseline | Kueue 실험 |
| --- | --- | --- |
| Redis queue | 빠르게 0 | 빠르게 0 |
| worker | queue 소비 후 감소 | queue 소비 후 감소 |
| 초과 workload 위치 | Kubernetes Pending Pod | Kueue LocalQueue / suspended Job |
| runner Pending Pod peak | 약 70개 이상 관찰 | 6 |
| admission 제어 | 없음 | ClusterQueue quota 기준 |
| admitted workload 수 | 제한 없음 | 약 8개로 제한 |

따라서 이번 실험은 성공으로 볼 수 있습니다.

Kueue가 전체 작업을 더 빨리 끝내는 도구는 아닙니다. 오히려 quota에 따라 실행량을 제한하므로 전체 완료 시간은 길어질 수 있습니다. 하지만 무제한으로 Pod를 만들어 scheduler 앞에 쌓는 대신, 실행 가능한 만큼만 admit하고 나머지는 Kueue queue에 보관합니다. 이번 실험에서 확인하고 싶었던 지점은 바로 이 부분이었습니다.

## 해석

이번 결과는 다음을 보여줍니다.

첫째, Redis + KEDA는 worker scale-out에는 유효하지만 runner Job admission control에는 부족합니다.

KEDA는 queue length를 기준으로 worker Pod를 늘리고 줄입니다. 하지만 worker가 queue를 모두 소비해 Job을 만들어버리면, 이후 backlog는 Redis queue가 아니라 Kubernetes Job/Pod 영역으로 넘어갑니다. KEDA는 이 상태를 직접 알지 못합니다.

둘째, Kueue는 backlog 위치를 더 적절한 곳으로 옮깁니다.

Kueue를 붙이면 초과 workload가 Pending Pod로 폭증하기 전에 Workload queue와 suspended Job으로 대기합니다. 이는 scheduler/API server 부담을 줄이고, 사용자 작업의 admission 상태를 더 명확하게 관찰할 수 있게 합니다.

셋째, Kueue quota는 실제 node capacity와 맞춰 튜닝해야 합니다.

이번 설정에서는 quota상 8개 workload가 admitted되었지만, 실제 runner Running Pod peak는 5였고 일부 admitted Pod는 Pending으로 남았습니다. 이는 ClusterQueue quota `4 CPU`가 현재 k3s 클러스터의 실제 배치 가능량보다 조금 높을 수 있음을 뜻합니다. 다음 실험에서는 quota를 `2.5` 또는 `3` CPU 정도로 낮추어 Pending Pod가 더 줄어드는지 볼 필요가 있습니다.

## 현재 아키텍처 변화

현재 단계의 아키텍처는 "Redis + KEDA를 Kueue가 완전히 대체한 구조"가 아닙니다.

정확히는 다음 구조입니다.

```text
API
-> Redis runs:queue
-> KEDA
-> worker
-> Kueue-managed Job
-> Kueue admission
-> Kubernetes scheduler
-> runner Pod
```

즉 Kueue는 기존 실행 경로 뒤쪽에 admission layer로 추가되었습니다.

현재 Redis와 KEDA는 여전히 다음 역할을 합니다.

- Redis는 API 요청과 run_id queue를 저장합니다.
- KEDA는 Redis queue 길이를 보고 worker를 깨웁니다.
- worker는 Redis queue를 소비해 Job을 만듭니다.
- Kueue는 만들어진 Job을 바로 실행할지, queue에서 기다리게 할지 결정합니다.

이 방식은 롤백이 쉽고 기존 구현 변경량이 작다는 장점이 있습니다. 그래서 실험 단계에서는 적절한 선택입니다.

다만 장기적으로는 execution queue를 Redis와 KEDA에 계속 둘 필요가 줄어듭니다. Kueue 검증이 충분히 끝나면 API가 직접 Kueue-managed Job을 생성하고, Redis queue와 worker를 Job 제출 경로에서 제거하는 구조를 검토할 수 있습니다.

장기 구조는 다음에 가깝습니다.

```text
API
-> Kueue-managed Job 생성
-> Kueue admission
-> Kubernetes scheduler
-> runner Pod

API
-> 별도 metadata store에 실행 기록 저장
```

이때 Redis를 계속 사용할지 여부는 별도 판단입니다. 단순 queue 용도라면 제거할 수 있습니다. 실행 기록, 사용자별 이력, 검색, 상태 관리까지 고려한다면 Redis보다 PostgreSQL 같은 영속 metadata store가 더 자연스러울 수 있습니다.

## 다음 개선 방향

이번 실험 이후 개선 방향은 다음과 같습니다.

1. ClusterQueue quota 튜닝

현재 quota `4 CPU`에서는 admitted 8개 중 일부 Pod가 Pending으로 남았습니다. 실제 노드 용량과 runner nodeSelector 조건을 고려해 `3 CPU` 또는 그 이하로 낮추어 다시 실험할 필요가 있습니다.

2. Kueue 상태를 API/UI에서 노출

현재 UI는 Redis queue, worker, runner Pod 중심으로 상태를 보여줍니다. Kueue를 도입하면 다음 값도 사용자 또는 운영자에게 중요해집니다.

- LocalQueue pending workloads
- LocalQueue admitted workloads
- Workload admitted 여부
- Job suspend 상태
- ClusterQueue quota 사용량

3. Redis queue 제거 가능성 검토

현재는 안전한 실험을 위해 Redis + KEDA + worker를 유지했습니다. 하지만 Kueue가 안정적으로 admission control을 수행한다면, 다음 단계에서는 API가 직접 Kueue-managed Job을 생성하는 구조를 검토할 수 있습니다.

4. GPU ResourceFlavor 확장

이번 실험은 CPU 기반 사전 검증입니다. 최종 목표가 GPU 노드 자동 확장 및 축소라면 이후에는 GPU resource request, GPU ResourceFlavor, GPU ClusterQueue quota를 도입해야 합니다.

5. Metric pipeline과 node autoscaling 연계

Kueue는 Job admission을 담당하고, Prometheus는 관찰을 담당하며, Karpenter는 node provisioning을 담당합니다. 장기적으로는 metric pipeline을 통해 queue 대기량, GPU 사용률, idle node, Pending workload를 관찰하고 Kueue quota 또는 Karpenter 정책과 연결하는 방향으로 발전시킬 수 있습니다.

## 결론

이번 Kueue 비교 실험의 핵심 성과는 기존 Redis + KEDA 방식의 병목 위치를 명확히 확인했다는 점입니다.

Redis queue가 0이 되었다고 해서 클러스터의 실행 backlog가 사라진 것은 아니었습니다. 기존 구조에서는 그 backlog가 Pending Pod로 밀려났고, 이는 scheduler/API server와 노드 자원 측면에서 부담이 될 수 있었습니다.

Kueue를 도입하자 초과 workload는 Pending Pod가 아니라 Kueue LocalQueue와 suspended Job으로 이동했습니다. 80 + 30 실험에서 runner Pending Pod peak가 크게 줄었고, admitted workload 수가 ClusterQueue quota 기준으로 제한되는 것도 확인했습니다.

따라서 현재 단계에서는 Kueue를 Redis + KEDA의 완전 대체재로 바로 보는 것보다, Job admission control을 담당하는 핵심 구성 요소로 도입하는 것이 타당합니다. 이후 검증이 더 쌓이면 Redis queue와 KEDA worker 경로를 줄이고, API가 직접 Kueue-managed Job을 생성하는 구조로 단순화할 수 있습니다.

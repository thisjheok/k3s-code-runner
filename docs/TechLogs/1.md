# VM 3대로 k3s 클러스터 구축하기

## 들어가며

이번 프로젝트에서는 사용자가 제출한 코드를 Kubernetes `Job`으로 실행하는 Mini Code Runner를 만들었습니다.

사용자가 코드를 제출하면 API 서버가 요청을 받고, 실행 작업은 별도의 worker가 처리하며, 실제 코드는 Kubernetes Job과 Pod로 격리해서 실행하는 구조를 실험하고 싶었습니다.

이를 위해서는 애플리케이션을 올릴 Kubernetes 환경이 필요했습니다. 다만 학습과 실험 목적의 프로젝트에서 처음부터 무거운 Kubernetes 클러스터를 구성하는 것은 부담이 컸습니다. 그래서 VM 3대를 기반으로 가볍게 운영할 수 있는 k3s 클러스터를 구성했습니다.

이번 글에서는 Mini Code Runner를 실행하기 위한 k3s 클러스터를 어떻게 구성했는지, 그리고 왜 그런 구조를 선택했는지 정리합니다.

## 전체 구성

클러스터는 총 3대의 VM으로 구성했습니다.

모든 VM에는 Linux Server 26.04를 설치했고, 역할은 다음과 같이 나누었습니다.

| 역할            | 대수  | 설명                             |
| ------------- | ---:| ------------------------------ |
| control plane | 1대  | k3s server가 실행되는 클러스터 제어 노드    |
| worker node   | 2대  | 애플리케이션 Pod와 코드 실행 Job이 배치되는 노드 |

구조를 단순화하면 다음과 같습니다.

```text
Windows 개발 환경
        |
        | kubectl / scp
        v
k3s cluster
  |
  |-- control plane VM
  |
  |-- worker node VM
  |
  |-- worker node VM
```

개발은 Windows 환경에서 진행하고, 실제 k3s 클러스터는 VM 위에서 실행했습니다. 프로젝트 파일은 필요할 때 VM으로 동기화한 뒤, VM 내부에서 이미지를 빌드하고 k3s에 배포하는 흐름으로 작업했습니다.

k3s는 설치가 비교적 단순하고 리소스 사용량이 가볍습니다. VM 몇 대만으로도 멀티 노드 Kubernetes 환경을 구성할 수 있고, 기본적으로 Traefik Ingress Controller도 포함되어 있어 로컬 도메인 기반 접근을 빠르게 구성할 수 있습니다.

## control plane 구성

첫 번째 VM은 control plane으로 사용했습니다.

control plane에는 k3s server를 설치하고, 클러스터의 Kubernetes API 서버 역할을 맡겼습니다. 이후 worker node들은 이 control plane에 join하는 방식으로 구성했습니다.

k3s 설치 후에는 VM에서 `kubectl`을 사용할 수 있도록 kubeconfig를 설정했습니다.

```bash
mkdir -p ~/.kube
sudo cp /etc/rancher/k3s/k3s.yaml ~/.kube/config
sudo chown $(id -u):$(id -g) ~/.kube/config
kubectl get nodes
```

`/etc/rancher/k3s/k3s.yaml`은 k3s가 생성하는 kubeconfig 파일입니다. 이를 일반 사용자 계정의 `~/.kube/config`로 복사해두면 이후 `kubectl` 명령을 매번 `sudo` 없이 실행할 수 있습니다.

설정이 완료되면 다음 명령으로 클러스터 노드 상태를 확인합니다.

```bash
kubectl get nodes
```

이 단계에서 control plane과 worker node들이 모두 `Ready` 상태가 되어야 이후 배포를 진행할 수 있습니다.

## worker node 구성

나머지 2대 VM은 worker node로 구성했습니다.

worker node는 control plane에 조인한 뒤, 실제 애플리케이션 Pod와 코드 실행 Job이 배치되는 실행 노드 역할을 합니다.

이번 프로젝트에서 worker node가 중요한 이유는 단순히 Pod를 더 많이 실행하기 위해서만은 아닙니다. Mini Code Runner는 사용자가 제출한 코드를 Kubernetes Job으로 실행합니다. 따라서 여러 실행 요청이 들어왔을 때 Job Pod들이 여러 노드에 분산될 수 있어야 합니다.

특히 이후 Redis와 KEDA를 붙이면 queue 길이에 따라 worker Pod 수가 늘어나고, worker Pod가 여러 Job을 생성하게 됩니다. 이때 멀티 노드 클러스터를 구성해두면 단일 노드 환경보다 실제 Kubernetes 스케줄링과 분산 실행을 더 현실적으로 관찰할 수 있습니다.

## 프로젝트 리소스를 namespace로 분리하기

Mini Code Runner의 Kubernetes 리소스는 관리 범위를 명확히 하기 위해서 `code-runner-system` namespace 안에 배치했습니다.

```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: code-runner-system
```

이 프로젝트에서는 API 서버, Redis, worker, Service, Ingress, RBAC, 그리고 실행 요청마다 생성되는 Job과 ConfigMap이 함께 사용됩니다. 이 리소스들이 `default` namespace에 섞이면 확인과 정리가 번거로워집니다.

반면 namespace를 분리하면 다음처럼 프로젝트 리소스만 좁혀서 확인할 수 있습니다.

```bash
kubectl -n code-runner-system get deploy,pod,svc
```

학습용 프로젝트라도 namespace를 분리해두면 이후 KEDA 리소스나 Job 리소스가 추가되어도 구조를 이해하기 쉬워집니다.

## 클러스터 위에 올라가는 구성 요소

k3s 클러스터 위에는 다음 구성 요소를 배포했습니다.

| 구성 요소  | Kubernetes 리소스                    | 역할                                             |
| ------ | --------------------------------- | ---------------------------------------------- |
| API 서버 | Deployment, Service, Ingress      | 사용자 요청을 받고 실행 상태를 조회합니다.                       |
| Redis  | Deployment, Service, PVC          | 실행 요청 queue와 실행 메타데이터를 저장합니다.                  |
| worker | Deployment                        | Redis queue를 소비하고 Kubernetes Job을 생성합니다.       |
| runner | Job, Pod, ConfigMap               | 사용자가 제출한 코드를 실제로 실행합니다.                        |
| RBAC   | ServiceAccount, Role, RoleBinding | API/worker가 Kubernetes 리소스를 생성하고 조회할 수 있게 합니다. |

전체 흐름은 다음과 같습니다.

```text
Browser UI
  -> code-runner-api Deployment
  -> Redis runs:queue
  -> code-runner-worker Deployment
  -> Kubernetes Job per run
  -> runner Pod
```

## RBAC 설정

이 프로젝트의 API 서버와 worker는 Kubernetes API를 직접 사용합니다.

Go 코드에서는 먼저 in-cluster config를 사용하도록 되어 있습니다.

```go
cfg, err := rest.InClusterConfig()
if err == nil {
    return kubernetes.NewForConfig(cfg)
}
```

즉, 애플리케이션이 Pod 안에서 실행될 때는 Kubernetes ServiceAccount 권한으로 API 서버에 접근합니다.

worker는 사용자 코드 실행을 위해 ConfigMap과 Job을 생성해야 합니다. API 서버는 실행 상태와 로그를 조회하기 위해 Pod, Job, Deployment 정보를 읽어야 합니다. 그래서 다음 리소스에 대한 권한을 부여했습니다.

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: code-runner-api
  namespace: code-runner-system
---
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: code-runner-role
  namespace: code-runner-system
rules:
  - apiGroups: [""]
    resources: ["configmaps"]
    verbs: ["get", "list", "create", "delete"]
  - apiGroups: [""]
    resources: ["pods"]
    verbs: ["get", "list", "watch"]
  - apiGroups: [""]
    resources: ["pods/log"]
    verbs: ["get"]
  - apiGroups: ["batch"]
    resources: ["jobs"]
    verbs: ["get", "list", "watch", "create", "delete"]
  - apiGroups: ["apps"]
    resources: ["deployments"]
    verbs: ["get", "list", "watch"]
```

권한 범위는 namespace 내부로 제한했습니다. ClusterRole이 아니라 Role을 사용한 것도 같은 이유입니다. Mini Code Runner가 필요한 리소스는 `code-runner-system` namespace 안에 있으므로, 클러스터 전체 권한을 줄 필요가 없었습니다.

## nodeSelector로 실행 위치 제어하기

사용자 코드 실행 Pod는 특정 worker node에 배치할 수 있도록 구성했습니다.

먼저 실행 노드로 사용할 worker node에 label을 부여합니다.

```bash
kubectl label node worker1 runner=true
```

그리고 API 서버와 worker에는 다음 환경 변수를 설정했습니다.

```yaml
- name: RUNNER_NODE_SELECTOR
  value: runner=true
```

Go 코드에서는 이 값을 파싱해서 Kubernetes Job의 Pod template에 `nodeSelector`로 넣습니다.

```go
NodeSelector: s.cfg.NodeSelector
```

이렇게 하면 사용자가 제출한 코드를 실행하는 runner Pod는 `runner=true` label이 있는 노드에 스케줄링됩니다.

이 구조를 선택한 이유는 실행 위치를 분리하기 위해서입니다.

API 서버나 Redis 같은 기본 구성 요소와 사용자 코드 실행 Pod는 성격이 다릅니다. API 서버는 안정적으로 떠 있어야 하고, runner Pod는 요청마다 생성되고 사라지는 일회성 실행 단위입니다. 따라서 runner Pod를 특정 노드 그룹에 배치할 수 있도록 만들어두면 이후 리소스 격리나 확장 전략을 실험하기 쉬워집니다.

## API 서버와 worker 배치

API 서버 Deployment에는 다음과 같은 설정이 들어갑니다.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: code-runner-api
  namespace: code-runner-system
spec:
  replicas: 1
  template:
    spec:
      nodeSelector:
        kubernetes.io/hostname: k3shandson
      serviceAccountName: code-runner-api
      containers:
        - name: api
          image: code-runner-api:latest
          imagePullPolicy: IfNotPresent
```

worker Deployment도 같은 이미지를 사용하지만, 실행 command만 다릅니다.

```yaml
containers:
  - name: worker
    image: code-runner-api:latest
    command:
      - /code-runner-worker
```

Dockerfile에서는 하나의 이미지 안에 API 서버 바이너리와 worker 바이너리를 함께 넣습니다.

```dockerfile
RUN CGO_ENABLED=0 GOOS=linux go build -o /out/code-runner-api ./cmd/api
RUN CGO_ENABLED=0 GOOS=linux go build -o /out/code-runner-worker ./cmd/worker
```

이렇게 구성하면 이미지 빌드와 배포 단위는 하나로 유지하면서, Kubernetes Deployment에서는 command를 바꿔 API 서버와 worker를 각각 실행할 수 있습니다.

worker의 replica 수는 처음에 `0`으로 설정했습니다.

```yaml
spec:
  replicas: 0
```

이 설정은 이후 KEDA가 Redis queue 길이를 보고 worker replica 수를 조절하도록 하기 위한 준비입니다. 기본 상태에서는 worker를 띄우지 않고, 실행 요청이 queue에 쌓이면 KEDA가 worker를 늘리는 구조로 확장할 수 있습니다.

## Service와 Ingress 구성

API 서버는 클러스터 내부에서 `ClusterIP` Service로 노출했습니다.

```yaml
apiVersion: v1
kind: Service
metadata:
  name: code-runner-api
  namespace: code-runner-system
spec:
  type: ClusterIP
  selector:
    app: code-runner-api
  ports:
    - name: http
      port: 80
      targetPort: 8080
```

브라우저에서 접근할 때는 Ingress를 사용했습니다.

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: code-runner-api
  namespace: code-runner-system
spec:
  rules:
    - host: code-runner.local
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: code-runner-api
                port:
                  number: 80
```

k3s는 기본적으로 Traefik Ingress Controller를 포함하고 있으므로, 별도 Ingress Controller 설치 없이 Ingress 리소스를 사용할 수 있습니다.

Windows 브라우저에서 `code-runner.local`로 접근하기 위해 hosts 파일에 VM IP를 등록했습니다.

```text
<VM_IP> code-runner.local
```

hosts 파일은 일반적으로 다음 경로에 있습니다.

```text
C:\Windows\System32\drivers\etc\hosts
```

이후 브라우저에서 다음 주소로 접근할 수 있습니다.

```text
http://code-runner.local/
```

## Redis 배포

Redis는 실행 요청 queue와 실행 메타데이터 저장소로 사용했습니다.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: code-runner-redis
  namespace: code-runner-system
spec:
  replicas: 1
```

Redis는 내부 통신용 `ClusterIP` Service로 노출했습니다.

```text
code-runner-redis.code-runner-system.svc.cluster.local:6379
```

API 서버와 worker는 환경 변수로 Redis 주소를 주입받습니다.

```yaml
- name: REDIS_ADDR
  value: code-runner-redis:6379
- name: REDIS_QUEUE_NAME
  value: runs:queue
```

Redis는 다음 글들에서 더 중요한 역할을 합니다. Go API 서버는 실행 요청을 Redis에 저장하고, worker는 Redis queue에서 실행 요청을 꺼내 Kubernetes Job을 생성합니다. KEDA는 같은 Redis queue 길이를 보고 worker replica 수를 조절합니다.

## 마무리

이번 글에서는 VM 3대를 기반으로 Mini Code Runner를 실행할 k3s 클러스터를 구성한 과정을 정리했습니다.

이 클러스터는 단순히 애플리케이션을 띄우는 공간이 아니라, 사용자가 제출한 코드를 Kubernetes Job으로 실행하고, worker를 통해 실행 요청을 처리하며, 이후 KEDA 기반 오토스케일링까지 실험하기 위한 기반입니다.

이번 구축을 통해 다음과 같은 기반을 마련했습니다.

- control plane 1대와 worker node 2대로 구성된 k3s 클러스터
- `code-runner-system` namespace 기반 리소스 분리
- Kubernetes API 접근을 위한 ServiceAccount와 RBAC
- 사용자 코드 실행 Pod 배치를 위한 node label과 nodeSelector
- API 서버 접근을 위한 Service와 Ingress
- Redis queue와 worker 구조를 올릴 수 있는 배포 기반

다음 글에서는 이 클러스터 위에서 동작하는 Go API 서버를 다룹니다. 사용자가 코드를 제출했을 때 API 서버가 요청을 어떻게 처리하고, Redis와 Kubernetes API를 어떻게 사용하는지 살펴보겠습니다.

# Redis와 KEDA로 worker 오토스케일링 구현하기

## 들어가며

이번 글에서는 Mini Code Runner의 세 번째 주제인 Redis와 KEDA 기반 오토스케일링을 다룹니다.

이 프로젝트에서 오토스케일링의 대상은 사용자가 제출한 코드를 직접 실행하는 runner Pod가 아닙니다. 정확히는 Redis queue를 소비하고 Kubernetes Job을 생성하는 `code-runner-worker` Deployment입니다. 실행 요청이 많아지면 worker Pod가 늘어나고, worker Pod가 queue를 병렬로 소비하면서 Kubernetes Job 생성 속도가 올라갑니다. queue가 비면 worker는 다시 0개까지 줄어듭니다.

즉, 이번 글의 핵심은 다음 흐름입니다.

```text
runs:queue length increases
-> KEDA increases code-runner-worker replicas
-> worker Pods consume the queue in parallel
-> each worker creates Kubernetes Jobs
-> runner Pods execute submitted code
-> queue drains
-> KEDA scales worker replicas back to 0
```

## 왜 오토스케일링이 필요한가

Mini Code Runner는 사용자가 제출한 코드를 Kubernetes Job으로 실행합니다. 단일 요청만 처리한다면 worker Pod 하나로도 충분합니다. 하지만 여러 사용자가 동시에 코드를 제출하거나, 테스트를 위해 짧은 시간에 많은 실행 요청을 보내면 상황이 달라집니다.

API 서버가 요청을 모두 직접 처리하면 요청량이 많아질수록 API 서버가 Kubernetes Job 생성을 계속 담당해야 합니다. 또한 실행 요청이 몰렸을 때 어느 정도의 backlog가 생겼는지, 처리량을 어떻게 늘릴지 판단하기 어렵습니다.

그래서 실행 요청을 Redis queue에 쌓고, worker가 queue를 소비하는 구조로 나누었습니다.

이 구조에서는 Redis queue 길이가 곧 처리 대기 중인 작업량입니다. queue 길이가 늘어나면 worker를 늘리고, queue가 비면 worker를 줄이면 됩니다. 이때 queue 길이를 직접 감시해서 Kubernetes Deployment replica 수를 조절하는 도구로 KEDA를 사용했습니다.

## 전체 구조

Redis와 KEDA를 포함한 전체 구조는 다음과 같습니다.

```text
Browser UI
  -> Go API Server
  -> Redis
     - run:{run_id}
     - runs:recent
     - runs:queue
  -> KEDA watches runs:queue length
  -> code-runner-worker Deployment scales out
  -> worker Pod dequeues run_id
  -> worker creates ConfigMap and Kubernetes Job
  -> runner Pod executes submitted code
```

각 구성 요소의 역할은 다음과 같습니다.

| 구성 요소 | 역할 |
| --- | --- |
| Go API Server | 실행 요청을 받고 Redis에 저장합니다. |
| Redis | 실행 메타데이터와 queue를 저장합니다. |
| KEDA | Redis queue 길이를 보고 worker Deployment를 스케일링합니다. |
| worker | Redis queue를 소비하고 Kubernetes Job을 생성합니다. |
| Kubernetes Job | 사용자 코드를 실제 runner Pod에서 실행합니다. |

이 구조에서 중요한 점은 API 서버, worker, runner의 책임이 분리되어 있다는 것입니다.

API 서버는 요청을 빠르게 접수합니다. worker는 실행 요청을 처리 가능한 속도로 소비합니다. runner Pod는 사용자 코드 실행이라는 일회성 작업만 담당합니다.

## Redis를 queue로 사용하기

API 서버는 `POST /runs` 요청을 받으면 실행 정보를 Redis에 저장합니다.

Redis에는 세 가지 key를 사용합니다.

| Redis key | 자료구조 | 역할 |
| --- | --- | --- |
| `run:{run_id}` | Hash | 실행 언어, 코드, timeout, 상태, 생성 시각을 저장합니다. |
| `runs:recent` | List | 최근 실행 목록을 UI/API에서 보여주기 위해 사용합니다. |
| `runs:queue` | List | worker가 소비할 실행 대기열입니다. |

실행 요청 생성 시점의 저장 로직은 다음과 같습니다.

```go
pipe := s.client.TxPipeline()
pipe.HSet(ctx, key, map[string]any{
    "run_id":          record.RunID,
    "language":        record.Language,
    "code":            record.Code,
    "timeout_seconds": strconv.FormatInt(record.TimeoutSeconds, 10),
    "status":          record.Status,
    "created_at":      record.CreatedAt,
})
pipe.LPush(ctx, s.recentRunsKey, record.RunID)
pipe.LTrim(ctx, s.recentRunsKey, 0, 99)
pipe.RPush(ctx, s.queueName, record.RunID)
_, err := pipe.Exec(ctx)
```

여기서 `RPush`로 `runs:queue`에 `run_id`를 넣습니다. worker는 이 queue에서 `run_id`를 꺼낸 뒤, `run:{run_id}` hash를 조회해 실제 실행 정보를 읽습니다.

이렇게 queue에는 무거운 실행 코드 전체를 넣지 않고 `run_id`만 넣었습니다. queue는 작업 순서와 대기량을 표현하고, 상세 데이터는 별도 hash에 보관하는 방식입니다.

이 구조는 다음 장점이 있습니다.

- queue 항목을 작게 유지할 수 있습니다.
- 실행 상세 정보는 `run:{run_id}`로 직접 조회할 수 있습니다.
- 최근 실행 목록과 실행 대기열을 분리할 수 있습니다.
- KEDA는 `runs:queue`의 길이만 보고 스케일링 판단을 할 수 있습니다.

## Redis 배포 구성

Redis는 Kubernetes Deployment로 배포했습니다.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: code-runner-redis
  namespace: code-runner-system
spec:
  replicas: 1
```

Redis container는 append-only file을 사용하도록 설정했습니다.

```yaml
args:
  - redis-server
  - --appendonly
  - "yes"
  - --dir
  - /data
```

그리고 `/data` 경로에는 PVC를 연결했습니다.

```yaml
volumeMounts:
  - name: redis-data
    mountPath: /data
```

학습용 프로젝트이지만 Redis가 queue와 실행 메타데이터를 함께 가지고 있으므로, Pod 재시작 시 데이터를 보존할 수 있도록 PVC를 사용했습니다.

클러스터 내부에서는 Redis Service를 통해 접근합니다.

```yaml
apiVersion: v1
kind: Service
metadata:
  name: code-runner-redis
  namespace: code-runner-system
spec:
  type: ClusterIP
  selector:
    app: code-runner-redis
  ports:
    - name: redis
      port: 6379
      targetPort: 6379
```

API 서버와 worker는 다음 주소로 Redis에 접근합니다.

```text
code-runner-redis:6379
```

KEDA ScaledObject에서는 namespace까지 포함한 service DNS를 사용했습니다.

```text
code-runner-redis.code-runner-system.svc.cluster.local:6379
```

## worker는 무엇을 하는가

worker는 Redis queue를 소비하고 Kubernetes Job을 만드는 프로세스입니다.

worker의 시작 지점은 `cmd/worker/main.go`입니다. API 서버와 마찬가지로 설정을 읽고, Kubernetes client와 Redis store를 준비합니다.

```go
cfg := config.Load()
clientset, err := k8s.NewClientset(cfg.KubeconfigPath)
store := runs.NewRedisStore(cfg.RedisAddr, cfg.RedisQueueName, cfg.RedisRecentRunsKey)
service := runs.NewService(clientset, store, cfg, logger)
```

그 다음 반복문 안에서 queue를 기다립니다.

```go
runID, err := store.DequeueRunID(ctx, 5*time.Second)
```

`DequeueRunID` 내부에서는 Redis `BLPop`을 사용합니다.

```go
result, err := s.client.BLPop(ctx, timeout, s.queueName).Result()
```

`BLPop`을 사용하면 queue가 비어 있을 때 worker가 바쁘게 polling하지 않고, 일정 시간 동안 blocking 상태로 대기할 수 있습니다.

worker가 `run_id`를 얻으면 Redis hash에서 실행 record를 읽습니다.

```go
record, err := store.GetRun(ctx, runID)
```

그리고 `SubmitRun`을 호출해 Kubernetes Job을 생성합니다.

```go
if err := service.SubmitRun(ctx, record); err != nil {
    logger.Error("failed to submit run", "run_id", runID, "err", err)
    continue
}
```

worker는 HTTP 요청을 받지 않습니다. 단지 queue를 기다리고, 작업이 들어오면 Kubernetes Job으로 변환하는 역할만 담당합니다.

## worker Deployment

worker는 별도 Deployment로 배포했습니다.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: code-runner-worker
  namespace: code-runner-system
spec:
  replicas: 0
```

초기 replica 수를 `0`으로 둔 것이 중요합니다.

worker는 항상 떠 있을 필요가 없습니다. 실행 요청이 없으면 queue도 비어 있고, worker가 할 일도 없습니다. 따라서 기본 상태에서는 worker를 0개로 두고, 요청이 생겼을 때 KEDA가 worker를 늘리도록 만들었습니다.

worker는 API 서버와 같은 이미지를 사용하지만, command를 override해 worker 바이너리를 실행합니다.

```yaml
containers:
  - name: worker
    image: code-runner-api:latest
    imagePullPolicy: IfNotPresent
    command:
      - /code-runner-worker
```

환경 변수는 API 서버와 거의 동일합니다.

```yaml
env:
  - name: RUNNER_NAMESPACE
    value: code-runner-system
  - name: RUNNER_TIMEOUT_SECONDS
    value: "10"
  - name: RUNNER_TTL_SECONDS
    value: "300"
  - name: RUNNER_NODE_SELECTOR
    value: runner=true
  - name: REDIS_ADDR
    value: code-runner-redis:6379
  - name: REDIS_QUEUE_NAME
    value: runs:queue
  - name: REDIS_RECENT_RUNS_KEY
    value: runs:recent
```

worker도 Kubernetes Job과 ConfigMap을 생성해야 하므로 API 서버와 같은 ServiceAccount를 사용합니다.

```yaml
serviceAccountName: code-runner-api
```

리소스 요청과 제한도 설정했습니다.

```yaml
resources:
  requests:
    cpu: 50m
    memory: 64Mi
  limits:
    cpu: 250m
    memory: 256Mi
```

worker 자체는 무거운 코드를 실행하지 않습니다. 사용자 코드는 별도 runner Pod에서 실행됩니다. 그래서 worker는 비교적 작은 리소스로 설정하고, 실제 사용자 코드 실행 리소스는 Job의 runner container에 따로 부여했습니다.

## worker가 만드는 Kubernetes Job

worker가 `SubmitRun`을 호출하면 먼저 사용자 코드를 ConfigMap으로 저장합니다.

```go
cm := &corev1.ConfigMap{
    ObjectMeta: metav1.ObjectMeta{
        Name:      name,
        Namespace: s.cfg.Namespace,
        Labels:    runLabels(record.RunID),
    },
    Data: map[string]string{spec.File: record.Code},
}
```

Python 코드는 `main.py`, Node.js 코드는 `main.js`로 저장됩니다.

그 다음 Kubernetes Job을 생성합니다.

```go
job := s.buildJob(name, record.RunID, spec, timeout)
```

Job에는 몇 가지 중요한 설정이 들어갑니다.

```go
BackoffLimit:            &backoffLimit,
ActiveDeadlineSeconds:   &timeout,
TTLSecondsAfterFinished: &s.cfg.TTLSeconds,
```

`BackoffLimit`은 0으로 설정했습니다. 사용자 코드가 실패했을 때 같은 코드를 재시도해도 성공 가능성이 높지 않고, 오히려 불필요한 리소스를 사용할 수 있기 때문입니다.

`ActiveDeadlineSeconds`는 실행 timeout입니다. 무한 루프나 오래 걸리는 코드가 Pod를 계속 점유하지 못하도록 제한합니다.

`TTLSecondsAfterFinished`는 완료된 Job을 일정 시간 후 정리하기 위한 설정입니다. 실행 결과를 잠시 조회할 수 있게 하되, 완료된 Job이 계속 쌓이지 않도록 했습니다.

runner container에는 리소스 제한도 설정했습니다.

```go
Resources: corev1.ResourceRequirements{
    Requests: corev1.ResourceList{
        corev1.ResourceCPU:    resource.MustParse("100m"),
        corev1.ResourceMemory: resource.MustParse("64Mi"),
    },
    Limits: corev1.ResourceList{
        corev1.ResourceCPU:    resource.MustParse("500m"),
        corev1.ResourceMemory: resource.MustParse("256Mi"),
    },
}
```

또한 보안 설정을 넣었습니다.

```go
SecurityContext: &corev1.SecurityContext{
    AllowPrivilegeEscalation: ptr(false),
    ReadOnlyRootFilesystem:   ptr(true),
    RunAsNonRoot:             ptr(true),
    RunAsUser:                ptr(int64(1000)),
    Capabilities:             &corev1.Capabilities{Drop: []corev1.Capability{"ALL"}},
}
```

사용자 코드를 실행하는 Pod이므로 최소한의 제한을 걸어두는 것이 중요했습니다. 이 설정만으로 완전한 sandbox가 되는 것은 아니지만, 학습용 코드 실행 환경에서 기본적인 격리 방향을 잡기 위한 선택이었습니다.

## KEDA를 선택한 이유

Kubernetes 기본 HPA는 CPU나 memory 같은 리소스 지표를 기준으로 replica 수를 조절하는 데 적합합니다.

worker가 바쁜지보다 중요한 것은 Redis queue에 처리 대기 중인 실행 요청이 얼마나 쌓였는가였습니다.

예를 들어 worker replica가 0개인 상태에서는 CPU 사용률이 존재하지 않습니다. 하지만 Redis queue에는 실행 요청이 쌓일 수 있습니다. 이때 queue 길이를 기준으로 0에서 worker를 깨워야 합니다.

KEDA는 이런 event-driven autoscaling에 적합합니다. Redis, Kafka, RabbitMQ 같은 외부 시스템의 지표를 보고 Kubernetes Deployment를 scale out/in할 수 있고, `minReplicaCount: 0` 구성이 가능합니다.

이번 프로젝트에서는 Redis list 길이를 기준으로 `code-runner-worker` Deployment를 스케일링했습니다.

## KEDA 설치

k3s 클러스터에 KEDA를 설치합니다.

```bash
kubectl apply --server-side -f https://github.com/kedacore/keda/releases/download/v2.19.0/keda-2.19.0.yaml
```

설치 후 namespace와 Deployment 상태를 확인합니다.

```bash
kubectl get ns keda
kubectl -n keda rollout status deployment/keda-operator
kubectl -n keda rollout status deployment/keda-metrics-apiserver
```

KEDA는 크게 operator와 metrics adapter 역할을 수행합니다. operator는 `ScaledObject` 같은 KEDA 리소스를 감시하고, metrics apiserver는 Kubernetes HPA가 사용할 수 있는 external metrics를 제공합니다.

## ScaledObject 구성

worker를 Redis queue 길이에 따라 확장하기 위해 `ScaledObject`를 작성했습니다.

```yaml
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: code-runner-worker-redis-queue
  namespace: code-runner-system
spec:
  scaleTargetRef:
    name: code-runner-worker
  pollingInterval: 5
  cooldownPeriod: 30
  minReplicaCount: 0
  maxReplicaCount: 10
  triggers:
    - type: redis
      metadata:
        address: code-runner-redis.code-runner-system.svc.cluster.local:6379
        listName: runs:queue
        listLength: "1"
```

핵심 설정은 다음과 같습니다.

| 설정 | 값 | 의미 |
| --- | --- | --- |
| `scaleTargetRef.name` | `code-runner-worker` | 스케일링할 대상 Deployment입니다. |
| `pollingInterval` | `5` | 5초마다 Redis queue 길이를 확인합니다. |
| `cooldownPeriod` | `30` | scale down 전에 30초 대기합니다. |
| `minReplicaCount` | `0` | 요청이 없으면 worker를 0개까지 줄입니다. |
| `maxReplicaCount` | `10` | worker를 최대 10개까지 늘립니다. |
| `type` | `redis` | Redis scaler를 사용합니다. |
| `listName` | `runs:queue` | 감시할 Redis list입니다. |
| `listLength` | `1` | worker 1개당 처리할 queue 길이 목표입니다. |

`listLength: "1"`은 매우 공격적인 설정입니다. queue에 작업이 1개만 있어도 worker를 늘릴 수 있습니다. 학습용 프로젝트에서는 오토스케일링 변화를 눈으로 빠르게 확인하는 것이 중요했기 때문에 이 값을 작게 잡았습니다.

운영 환경이라면 작업 처리 시간, worker 시작 시간, 평균 요청량, 비용 등을 고려해 더 완만한 값을 선택해야 합니다.

ScaledObject 적용은 다음 명령으로 수행합니다.

```bash
kubectl apply -f deploy/keda-scaledobject.yaml
```

적용 후 상태를 확인합니다.

```bash
kubectl -n code-runner-system get scaledobject
kubectl -n code-runner-system describe scaledobject code-runner-worker-redis-queue
```

## 오토스케일링 동작 흐름

전체 동작을 시간 순서로 보면 다음과 같습니다.

1. 사용자가 UI 또는 curl로 실행 요청을 보냅니다.
2. API 서버가 Redis `run:{run_id}`에 실행 정보를 저장합니다.
3. API 서버가 Redis `runs:queue`에 `run_id`를 넣습니다.
4. KEDA가 Redis `runs:queue` 길이를 확인합니다.
5. queue 길이가 목표값보다 크면 `code-runner-worker` replica 수를 늘립니다.
6. 새 worker Pod가 시작됩니다.
7. worker가 `BLPop`으로 `runs:queue`에서 `run_id`를 가져옵니다.
8. worker가 Redis에서 실행 record를 읽습니다.
9. worker가 ConfigMap과 Kubernetes Job을 생성합니다.
10. Job이 runner Pod를 만들고 사용자 코드를 실행합니다.
11. queue가 줄어듭니다.
12. cooldown 시간이 지난 뒤 KEDA가 worker replica 수를 다시 줄입니다.

이 구조에서 queue는 단순한 저장소가 아니라 시스템의 압력 지표입니다. queue가 길어질수록 처리해야 할 일이 많다는 뜻이고, queue가 0이면 worker를 유지할 이유가 없습니다.

## UI에서 오토스케일링 관찰하기

이번 프로젝트에서는 오토스케일링 흐름을 브라우저 UI에서 볼 수 있도록 `GET /system/status` API를 만들었습니다.

UI는 다음 값을 표시합니다.

- `queue_depth`
- `worker_replicas`
- worker Pod 목록
- runner Job 목록
- runner Pod 목록
- node별 runner Pod 분포
- queue와 worker replica 변화 timeline

프론트엔드에서는 `/system/status`를 호출해 상태를 갱신합니다.

```js
const status = await fetchJSON("/system/status");
```

그리고 queue 길이와 worker replica 수를 history에 쌓아 timeline으로 렌더링합니다.

```js
history.push({
  queueDepth,
  workerReplicas,
  at: Date.now(),
});
```

UI에는 `Batch count`와 `Sleep seconds` 입력값도 있습니다. 여러 실행 요청을 한 번에 보내고, 각 코드가 일정 시간 sleep하도록 만들어 queue와 worker replica 변화를 관찰할 수 있습니다.

테스트 시나리오는 다음과 같습니다.

1. `Batch count`를 `20`으로 설정합니다.
2. `Sleep seconds`를 `20`으로 설정합니다.
3. `Run` 버튼을 클릭합니다.
4. `queue_depth`, `worker_replicas`, node distribution, Live Monitor timeline을 확인합니다.

이 시나리오에서는 실행 요청이 한 번에 여러 개 들어가므로 `runs:queue` 길이가 증가합니다. KEDA가 이를 감지해 worker replica를 늘리고, worker들이 queue를 병렬로 소비하면서 runner Job을 생성합니다.

## curl로 테스트하기

UI 대신 curl로도 동일한 테스트를 할 수 있습니다.

```bash
for i in $(seq 1 20); do
  curl -s -X POST http://code-runner.local/runs \
    -H "Content-Type: application/json" \
    -d '{"language":"python","code":"import time\nprint(\"batch run\")\ntime.sleep(20)","timeout_seconds":60}' >/dev/null
done
```

worker Deployment 변화를 watch합니다.

```bash
kubectl -n code-runner-system get deploy code-runner-worker -w
```

queue 길이는 Redis에서 직접 확인할 수 있습니다.

```bash
kubectl -n code-runner-system exec deploy/code-runner-redis -- redis-cli llen runs:queue
```

worker 로그도 확인합니다.

```bash
kubectl -n code-runner-system logs deploy/code-runner-worker --tail=50
```

queue가 비고 cooldown 시간이 지나면 worker replica 수가 다시 0으로 내려가야 합니다.

```bash
kubectl -n code-runner-system get deploy code-runner-worker
kubectl -n code-runner-system exec deploy/code-runner-redis -- redis-cli llen runs:queue
```

## 멀티 노드에서 runner Pod 분산 확인하기

1편에서 k3s 클러스터를 3대 VM으로 구성한 이유 중 하나는 runner Pod가 여러 node에 배치되는 흐름을 확인하기 위해서였습니다.

runner Pod는 `RUNNER_NODE_SELECTOR` 값을 통해 특정 label이 있는 node에 배치됩니다.

```bash
kubectl label node worker1 runner=true
```

Deployment에는 다음 환경 변수가 들어갑니다.

```yaml
- name: RUNNER_NODE_SELECTOR
  value: runner=true
```

worker가 Job을 만들 때 이 값은 Pod template의 `nodeSelector`로 들어갑니다.

```go
NodeSelector: s.cfg.NodeSelector
```

따라서 runner Pod는 `runner=true` label이 붙은 node에 스케줄링됩니다.

UI의 Live Monitor에서는 node별 runner Pod 수를 볼 수 있습니다. API 서버는 runner Pod 목록을 조회하면서 `pod.Spec.NodeName`을 기준으로 집계합니다.

```go
nodeRuns := map[string]int{}
...
nodeRuns[nodeName]++
```

이를 통해 단순히 worker replica가 늘어나는 것뿐 아니라, 실제 코드 실행 Pod가 어느 node에 배치되었는지도 확인할 수 있습니다.

## 이 구조의 장점

Redis와 KEDA를 사용한 구조의 가장 큰 장점은 API 서버와 실행 처리량을 분리할 수 있다는 점입니다.

API 서버는 요청을 빠르게 Redis에 저장하고 응답합니다. 실행 요청이 많아져도 queue에 쌓을 수 있고, worker 수는 queue 길이에 따라 독립적으로 조절됩니다.

또한 `minReplicaCount: 0`을 사용할 수 있다는 점도 중요했습니다. 실행 요청이 없는 동안 worker Pod를 유지하지 않아도 됩니다. 학습용 VM 환경에서는 리소스가 제한적이기 때문에, 사용하지 않는 worker를 0개로 줄일 수 있는 구조가 잘 맞았습니다.

그리고 queue 길이가 명확한 관찰 지표가 됩니다. CPU 사용률만 보면 지금 처리 대기 중인 실행 요청이 얼마나 밀렸는지 알기 어렵습니다. 반면 `runs:queue` 길이는 backlog를 직접 보여줍니다.


## 한계와 개선할 점

이번 구현은 학습과 구조 검증에 초점을 맞췄습니다. 따라서 운영 수준으로 확장하려면 몇 가지 개선이 필요합니다.

첫째, Redis는 단일 Deployment로 구성되어 있습니다. 실서비스라면 Redis 장애에 대비한 고가용성 구성이 필요합니다.

둘째, queue 소비 후 Job 생성에 실패한 경우 재처리 전략이 더 정교해야 합니다. 현재 worker는 `BLPop`으로 queue에서 항목을 꺼낸 뒤 처리합니다. Job 생성에 실패하면 상태를 실패로 남길 수 있지만, 메시지를 다시 queue에 넣거나 dead letter queue로 보내는 전략은 아직 없습니다.

셋째, 사용자 코드 실행 sandbox를 더 강화해야 합니다. 현재 runner Pod에는 non-root, read-only root filesystem, capability drop, resource limit을 적용했지만, 네트워크 제한이나 더 강한 격리까지 포함한 완전한 실행 sandbox는 아닙니다.

넷째, `listLength: "1"`은 관찰에는 좋지만 운영 기준으로는 조정이 필요합니다. worker 시작 시간, 평균 Job 생성 속도, queue 유입량을 측정한 뒤 적절한 목표값을 정해야 합니다.

## 마무리

이번 글에서는 Redis와 KEDA를 이용해 Mini Code Runner의 worker를 오토스케일링하는 구조를 살펴봤습니다.

API 서버는 실행 요청을 Redis에 저장하고 `runs:queue`에 `run_id`를 넣습니다. KEDA는 같은 Redis list의 길이를 감시하고, queue가 쌓이면 `code-runner-worker` Deployment를 scale out합니다. worker는 queue를 소비해 Kubernetes Job을 생성하고, Job이 runner Pod를 통해 사용자 코드를 실행합니다. queue가 비면 KEDA는 worker를 다시 0개까지 줄입니다.

이번 구현을 통해 다음 흐름을 만들 수 있었습니다.

- Redis list 기반 실행 queue
- worker의 `BLPop` 기반 queue 소비
- KEDA Redis scaler 기반 worker scale-out/scale-in
- Kubernetes Job 기반 사용자 코드 실행
- UI Live Monitor를 통한 queue, worker, runner Pod, node 분포 관찰

이 구조는 작은 프로젝트이지만, event-driven autoscaling의 핵심 흐름을 직접 확인하기에 좋은 예제였습니다. 특히 “요청을 queue에 쌓고, queue 길이를 기준으로 처리자를 늘린다”는 패턴은 코드 실행 시스템뿐 아니라 비동기 작업 처리 시스템 전반에 응용할 수 있는 구조입니다.
